# Agent Performance & Reliability Improvements

**Last Updated: 2026-03-22**

## Executive Summary

Агент медленный и не может выполнить сложные задачи (например, откликнуться на вакансию на hh.ru). Корневые причины:

1. **Неполная загрузка SPA** — `wait_for_page_ready` ждёт только `domcontentloaded`, SPA-сайты (React/Vue) не успевают гидрироваться
2. **Двойная извлечение DOM** — после `navigate`/`click` page_state извлекается дважды (в `_do_action` + в `execute_tool`)
3. **Бесполезные скриншоты** — quality=35 делает текст нечитаемым, агент не анализирует скриншоты через vision
4. **Слабый DOM-парсинг** — лимит 150 элементов, 4000 символов текста, нет кастомных кнопок (`div.btn`)
5. **Нулевая observability** — нет метрик по времени, нет логов скриншотов, нет отладки DOM
6. **Слабый системный промпт** — нет guidance по vision, слишком verbose примеры, нет hh.ru-специфики

## Current State Analysis

### Performance Bottlenecks

| Проблема | Влияние | Частота |
|----------|---------|---------|
| Двойная извлечение DOM (navigate → execute_tool) | +10-20% лишнего времени | Каждая навигация |
| Скриншот quality=35, но 150KB+ base64 | +50K токенов на скриншот | Каждая навигация + click |
| `wait_for_page_ready` только DOM, без гидрации | Агент видит неполный DOM SPA-сайтов | Каждая навигация на hh.ru |
| Лимит 150 элементов в парсере | Не находит элементы на больших страницах | Страницы списков, поиск |
| Текст обрезается на 4000 символов | Теряет контент на насыщенных страницах | hh.ru с множеством вакансий |
| План инжектится каждый шаг (100+ символов) | Bloat даже без изменений | Каждый шаг |

### Why hh.ru Tasks Fail

1. **Начальная загрузка неполная** — React гидрация не завершена → пустые списки вакансий
2. **Модальные окна блокируют клики** — overlay не обнаруживается, агент зацикливается
3. **Кастомные дропдауны** — `select_option` работает только с `<select>`, не с React-компонентами
4. **Бюджет исчерпывается** — к шагу 25-30 агент потратил 50%+ шагов на "разбирательство"
5. **Нет vision-анализа** — скриншоты передаются, но агент не может "увидеть" что на них

## Proposed Future State

После всех улучшений:
- **Время на шаг**: 2-3с → 1-2с (за счёт устранения дублей и оптимизации ожиданий)
- **Качество DOM**: агент видит полный, стабильный DOM даже на SPA-сайтах
- **Vision**: агент использует скриншоты для верификации (читаемое качество, explicit logging)
- **Observability**: полные метрики — время каждого инструмента, токены, передача скриншотов
- **Успешность hh.ru**: агент может пройти полный flow "поиск → открытие → отклик"

## Implementation Phases

### Phase 1: Observability & Logging (CRITICAL — без этого не видим что чинить)

Добавить инструментальные метрики, чтобы видеть bottleneck-и в реальном времени.

**Rationale**: Прежде чем оптимизировать, нужно измерить. Сейчас мы не знаем, сколько времени занимает каждый инструмент, когда передаётся скриншот, какой DOM видит агент.

### Phase 2: Performance — Wait Strategy & DOM Extraction

Ускорить агента, убрав дублирование и улучшив ожидание загрузки.

**Rationale**: Самые impactful изменения — убрать двойную извлечение DOM и сделать нормальный wait для SPA.

### Phase 3: DOM Quality — Parser Improvements

Улучшить качество DOM, который видит агент, чтобы он находил нужные элементы.

**Rationale**: Если агент не видит кнопку "Откликнуться", он не может на неё нажать.

### Phase 4: Vision & Screenshot Improvements

Сделать скриншоты полезными — повысить качество, логировать их передачу, использовать для верификации.

**Rationale**: Сейчас скриншоты есть, но бесполезны: quality=35 нечитаем, агент не анализирует их.

### Phase 5: System Prompt & Agent Intelligence

Переработать системный промпт для лучшего guidance и уменьшения overhead.

**Rationale**: Промпт ~900 токенов с verbose примерами. Нужно сделать его более actionable и добавить guidance по vision.

### Phase 6: Context & Loop Detection

Оптимизировать управление контекстом и детекцию зацикливания.

**Rationale**: 40 сообщений = ~20 циклов действий. Ранний контекст теряется без суммаризации.

## Risk Assessment

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Новый wait_for_page_ready слишком медленный | Средняя | Таймауты + fallback на старое поведение |
| Увеличение лимитов парсера → больше токенов | Высокая | Мониторить размер page_state в логах |
| Изменение промпта ломает текущее поведение | Средняя | Тестировать на 3-5 разных задачах до/после |
| Увеличение quality скриншотов → больше base64 | Низкая | detail="low" в API уменьшает token count |

## Success Metrics

1. **Время на шаг** < 2с (медиана) — измеряем через новые логи
2. **hh.ru flow**: агент проходит "navigate → поиск → открытие вакансии → отклик" за ≤15 шагов
3. **DOM completeness**: все интерактивные элементы hh.ru видны в page_state (верифицируем вручную)
4. **Vision usage**: в логах видно, когда агент получает скриншот + его размер
5. **Zero double-extraction**: в логах нет двух `extract_page_state` подряд на одну навигацию
